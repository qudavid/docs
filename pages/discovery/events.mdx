# List Events

> **GET `/v1/events`** ‚Äî Discover canonical events across venues. Each event can map to multiple venue-native IDs and anchor one or more markets.

- **Base URL:** `https://api.predexon.com/v1`
- **Auth:** `Authorization: Bearer <token>`
- **Time:** All timestamps are **UTC ISO-8601** (e.g., `2025-10-06T19:21:02Z`)
- **Pagination:** Cursor-based via `next_cursor` (no `prev_cursor`)

## Common use cases

- üèü **Sports on a date:** MLB games on `2025-07-12`
- üó≥ **Upcoming elections:** US elections with `event_status=scheduled`
- üìä **Macro releases:** Find CPI/Fed events for a date to anchor markets

## Endpoint

```http
GET /v1/events
```

## Query parameters

All are optional unless noted.

| Parameter | Type | Notes |
|-----------|------|-------|
| `category` | string | e.g. `sports/baseball/mlb`, `politics/elections/us`, `macro/cpi` |
| `event_status` | enum | `scheduled` \| `live` \| `resolved` \| `cancelled` |
| `date` | string | UTC date `YYYY-MM-DD`. Exact-match on the event's scheduled date in UTC. |
| `participant` | string | Simple participant code (e.g., `NYY`, `BOS`, `BIDEN`) |
| `event_uid` | string | Exact canonical ID match |
| `venue` | enum | `POLYMARKET` \| `KALSHI` |
| `polymarket_slug` | string | Match Polymarket slug |
| `kalshi_event_ticker` | string | Match Kalshi event ticker |
| `limit` | integer | Default 50, max 200 |
| `cursor` | string | Opaque cursor from the previous page's `next_cursor` |

<Info>
**Tip:** Prefer `category` + `date` for daily scans; use `event_status` to catch late changes (postponed/cancelled vs scheduled).
</Info>

## Field reference

| Field | Type | Description |
|-------|------|-------------|
| `event_uid` | string | Canonical, stable event identifier |
| `title` | string | Human-readable event title |
| `category` | string | Taxonomy path (e.g., `sports/baseball/mlb`) |
| `scheduled_start` | string (ISO-8601) | Scheduled UTC start timestamp |
| `event_status` | enum | Lifecycle: `scheduled`, `live`, `resolved`, `cancelled` |
| `participants` | string[] | Simple codes (teams, candidates, etc.) |
| `venue_ids.polymarket` | string[] | Polymarket market/event slugs referencing this event |
| `venue_ids.kalshi` | string[] | Kalshi event tickers referencing this event |
| `asof` | string (ISO-8601) | Snapshot capture time (useful for caching/checkpointing) |
| `next_cursor` | string | Cursor for the next page (omit when last page) |

## Examples

<CodeGroup>

```bash cURL
# 1) MLB events on a specific date
curl -s -H "Authorization: Bearer $PREDX_TOKEN" \
  "https://api.predexon.com/v1/events?category=sports/baseball/mlb&date=2025-07-12&limit=100"

# 2) US elections that are scheduled (not started)
curl -s -H "Authorization: Bearer $PREDX_TOKEN" \

# 3) Pagination: fetch the next page using next_cursor

  "https://api.predexon.com/v1/events?cursor=eyJpZCI6..."
```

```javascript JavaScript
// Node 18+ (fetch)
const base = "https://api.predexon.com/v1";
const headers = { Authorization: `Bearer ${process.env.PREDX_TOKEN}` };

async function listMlbByDate() {
  const url = new URL(`${base}/events`);
  url.searchParams.set("category", "sports/baseball/mlb");
  url.searchParams.set("date", "2025-07-12");
  url.searchParams.set("limit", "100");

  const res = await fetch(url, { headers });
  if (!res.ok) throw new Error(`${res.status} ${await res.text()}`);
  const data = await res.json();
  return data.events;
}
```

```python Python
import os, requests

BASE = "https://api.predexon.com/v1"

def list_us_elections_scheduled():
  params = {
    "category": "politics/elections/us",
```

```go Go

import (
```

## Pagination

The API returns `next_cursor` when more results are available.

```bash
# First page
curl -s -H "Authorization: Bearer $PREDX_TOKEN" \
  "https://api.predexon.com/v1/events?category=sports/baseball/mlb&date=2025-07-12&limit=100"

# Follow-up page
curl -s -H "Authorization: Bearer $PREDX_TOKEN" \
  "https://api.predexon.com/v1/events?cursor=eyJpZCI6..."
```

<Warning>
No `prev_cursor` in v1. Keep your last cursor to iterate forward only.
</Warning>

## Best practices

1. **Daily scans:** Use `category` + `date` and store `event_uid` + `asof`.

2. **Change detection:** Re-fetch with `event_status` filters to catch postponements/cancellations.

3. **Cursor loops:** Continue until `next_cursor` is absent.

4. **Venue hopping:** Use `venue_ids.*` to jump into venue-native lookups.

5. **Caching:** Cache by querystring and invalidate when `asof` changes.
