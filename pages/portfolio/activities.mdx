# Activities
<Callout type="warning" title="Experimental">
This endpoint group is experimental and may change without notice. Access is limited; contact support if you need early integration.
</Callout>

> **GET `/v1/portfolio/activities`** — Retrieve a unified ledger of trades, deposits, withdrawals, and resolutions across linked venues.

- **Base URL:** `https://api.predexon.com/v1`
- **Auth:** `Authorization: Bearer <token>`
- **Time:** All timestamps are **UTC ISO-8601**
- **Rate Limits:** 40 requests/minute per API key. `429` responses include `Retry-After` (seconds).

## Endpoint

```http
GET /v1/portfolio/activities
```

## Query parameters

All parameters are optional.

| Parameter | Type | Description |
|-----------|------|-------------|
| `type` | enum | Filter by activity type: `trade`, `deposit`, `withdrawal`, `resolution`. |
| `start` | string | Inclusive start timestamp filter (`ISO-8601`). |
| `end` | string | Exclusive end timestamp filter (`ISO-8601`). |
| `limit` | integer | Results per page (default 100, max 1000). |
| `cursor` | string | Opaque cursor returned in `next_cursor`. |

## Field reference

| Field | Type | Description |
|-------|------|-------------|
| `activities` | array | Ledger entries ordered newest first. |
| `activities[].ts` | string (ISO-8601) | Activity timestamp. |
| `activities[].type` | string | `trade` \| `deposit` \| `withdrawal` \| `resolution`. |
| `activities[].venue` | string | Venue code (present for venue-based activities). |
| `activities[].market_uid` | string \| null | Canonical market identifier (trades/resolutions). |
| `activities[].outcome` | string \| null | Outcome code (trades). |
| `activities[].price` | string \| null | Trade price. |
| `activities[].size` | string \| null | Trade size. |
| `activities[].fee` | string \| null | Trade fee (negative). |
| `activities[].id` | string \| null | Venue-specific identifier. |
| `activities[].amount` | string \| null | Cash amount (deposits/withdrawals). |
| `activities[].currency` | string \| null | Currency for `amount`. |
| `activities[].outcome_won` | string \| null | Winning outcome (resolutions). |
| `activities[].payout` | string \| null | Net payout for the resolution. |
| `next_cursor` | string \| null | Cursor for the next page (`null` when no more results). |

<Info>
**Activity types**

- **trade:** Executions affecting position size; includes price, size, fee, and venue.
- **deposit/withdrawal:** Cash movements; look at `amount`, `currency`, and `venue`.
- **resolution:** Market settlements; `payout` reflects net result (positive or negative) for the outcome you held.
</Info>

## Examples

<CodeGroup>

```bash cURL
curl -s "https://api.predexon.com/v1/portfolio/activities?type=trade&limit=200" \
  -H "Authorization: Bearer $PREDX_TOKEN"
```

```javascript JavaScript
const base = "https://api.predexon.com/v1";

async function listActivities(params = {}) {
  const url = new URL(`${base}/portfolio/activities`);
  Object.entries(params).forEach(([key, value]) => url.searchParams.set(key, value));

  const res = await fetch(url, { headers: { Authorization: `Bearer ${process.env.PREDX_TOKEN}` } });
  if (!res.ok) throw new Error(`${res.status} ${await res.text()}`);
  return res.json();
}
```

```python Python
import os
import requests

def list_activities(**params) -> dict:
  res = requests.get(
      "https://api.predexon.com/v1/portfolio/activities",
      headers={"Authorization": f"Bearer {os.environ['PREDX_TOKEN']}"},
      params=params,
      timeout=30,
  )
  res.raise_for_status()
  return res.json()

print(list_activities(type="deposit", limit=50))
```

## Errors

All errors return standard HTTP status codes with JSON error responses:

```json
{
  "error": {
    "type": "insufficient_permissions",
    "message": "link an account before requesting activities"
  }
}
```

| HTTP Status | Error Type | Description | Common Causes |
|-------------|------------|-------------|---------------|
| `400` | `invalid_parameter` | Request validation failed | Unknown `type`, invalid timestamp |
| `401` | `invalid_token` | Authentication failed | Missing or invalid `Authorization` header |
| `403` | `insufficient_permissions` | User has not linked required accounts | Link accounts first |
| `429` | `rate_limit_exceeded` | Too many requests | Exceeded 40 req/min |
| `500` | `internal_error` | Server error | Retry with backoff |
| `503` | `service_unavailable` | Temporary service issue | Retry with exponential backoff |

## Best practices

1. **Use cursors:** For high-activity accounts, iterate with `next_cursor` instead of raising `limit` excessively.
2. **Reconcile with PnL:** Tie realized PnL and fees from trades/resolutions back to [PnL](/pages/portfolio/pnl) curves.
3. **Filter by type:** Use the `type` parameter to drive activity-specific workflows (e.g., cash ledger vs trade log).
4. **Store IDs:** Persist `activities[].id` for idempotent processing and reconciliation with venue statements.
